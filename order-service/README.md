# Order Service - Real-Time Payment Settlement System (RTPS)

## Overview
The **Order Service** is the central entry point of the RTPS ecosystem. Built with Spring Boot 3.2, it is responsible for managing user accounts, authenticating requests via secure JWTs, and capturing payment orders with an absolute guarantee of idempotency, financial consistency, and reliability.

---

## 🚀 Implemented Concepts (Production Hardened)

### 1. Advanced API Design & Reliability
- **Transactional Outbox Pattern**: Ensures 100% reliable event delivery. Instead of publishing directly to Kafka, events are saved to the Oracle DB in the same transaction as the Order. A background `OutboxPublisher` job then guarantees delivery to Kafka, even if the broker is temporarily offline.
- **HTTP Header Idempotency**: Follows industry-standard API patterns (like OFSS/Stripe) by extracting the `Idempotency-Key` from the HTTP Headers. Includes strict validation (required, max 255 chars).
- **Rate Limiting**: Integrated **Resilience4j** with `@RateLimiter` annotation on the Order creation endpoint to prevent abuse and ensure fair resource allocation.
- **Intelligent Error Handling**: Standardized JSON responses for all errors. Includes a specific handler for **429 Too Many Requests** when rate limits are exceeded.
- **Server-Side Protection**: Configured Tomcat `connection-timeout` and `keep-alive-timeout` to protect system resources from slow or hanging client connections.

### 2. Distributed Tracing & Observability
- **End-to-End Tracing**: The `X-Correlation-ID` is automatically propagated from the HTTP request into the Kafka message payload. This allows for seamless distributed tracing across microservices.
- **Lombok `@Slf4j` Logging**: High-performance, clean logging implementation.
- **Correlation ID in Logs**: Every log line automatically includes the correlation ID in `[...]` brackets via MDC.
- **Spring Boot Actuator**: Exposes health, metrics, and info endpoints at `/actuator/**` for production monitoring.

### 3. User Management & Security
- **Secure Key Management**: Encryption keys and JWT secrets are externalized. In production, these are loaded from environment variables (`ENCRYPTION_KEY`, `JWT_SECRET`), while `application-local.yml` provides secure overrides for development.
- **JWT Refresh Token Rotation**: Implemented token rotation where every refresh call invalidates the old token and issues a new one to mitigate replay attacks.
- **Data Encryption at Rest**: Sensitive financial fields like `recipientBankAccount` are encrypted using **AES-256** before being saved to the Oracle DB.
- **Password Reset Flow**: Complete password reset functionality with time-limited tokens via `PasswordResetToken` entity.
- **Method-Level Security**: `@EnableMethodSecurity` configured for fine-grained access control.

### 4. Kafka Event-Driven Architecture
- **OrderCreatedEvent**: Published to Kafka when a new order is created (via Outbox Pattern).
- **PaymentEventConsumer**: Listens to `payment-events` topic to update order status based on payment processor results.
- **Internal Service Communication**: `PaymentProcessorClient` uses `RestTemplate` with `X-Internal-Secret` header for authenticated service-to-service calls.

### 5. Database Evolution & Persistence
- **Flyway Schema Migrations**: Fully automated schema versioning (V1 through V9). Uses modern Oracle features like `GENERATED BY DEFAULT AS IDENTITY`.
- **Profile-Driven Configuration**: Segregated "local" and "production" configurations to prevent credential leakage.

## 🔑 Key Features Detail

### Order Status Flow
```
PENDING → COMPLETED (on payment success)
PENDING → FAILED (on payment failure)
```

### Idempotency Key
- Stored in database with expiry
- Cleanup task runs periodically to remove expired keys
- Required header: `Idempotency-Key` (max 255 chars)

### Transactional Outbox
- Events saved in same transaction as order
- Background job polls every 10 seconds
- Exponential backoff: 1s, 2s, 4s, 8s... max 5 minutes
- Max 5 retries before marking as FAILED

### Security Headers
- CORS: `http://localhost:8000`, `http://localhost:3000`
- Allowed Headers: `Authorization`, `Content-Type`, `X-Correlation-ID`, `Idempotency-Key`
- Exposed Headers: `X-Correlation-ID`
